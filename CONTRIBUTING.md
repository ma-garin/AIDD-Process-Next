# CONTRIBUTING — AIDD Process Next

本書は AIDD Process Next への貢献手順を定めます。本プロジェクトは AI駆動開発の品質成熟度を10領域50問で自己診断する静的Webアプリ（検証前ドラフト v0.1）です。貢献の前に `agent.md`（運用方針）と `sqec.md`（受入条件・品質基準）を必ず確認してください。

リポジトリ: GitHub `ma-garin/AIDD-Process-Next`

---

## 1. 基本姿勢

- 本ツールは日本のB2B業務システムです。落ち着いたトーン、読みやすい日本語、説明責任を重視します。
- 変更は小さく、レビュー可能な単位で行います。
- スコア・設問は暫定値であり、リサーチゲート完了までは認証・監査基準として扱いません。

---

## 2. 非交渉原則（必読）

`agent.md` に基づく以下の原則は、いかなる貢献でも逸脱できません。

- TPI NEXT・QA4AIDD など第三者の専有文言・専有モデルの文章をそのまま流用しない。一般に公開された規格名・概念名（例: ISO/IEC 25010、ISTQB 用語）の参照は可とするが、文章の引き写しは不可。
- 外部に対して本モデルを「TPI」あるいは特定商標を冠した名称で呼称・表記しない。本モデルは独自の成熟度モデルである。
- 未検証のスコアを認証・保証であるかのように表現しない。
- バックエンド、認証、DB、AI API、外部CDN・外部JS依存を明示指示なく追加しない。
- 秘密情報・個人情報・認証情報をコードやサンプルへ含めない。
- 設問データをUIコンポーネントへ直書きしない。スコアリングと描画を混在させない。

疑わしい場合は採用せず、Issue で相談してください。

---

## 3. 設問改訂プロセス（本プロジェクトの中核）

設問（`docs/data/questions.js` の10領域50問）、重み、閾値、改善Issue案の改訂は、本ツールの品質に直結します。以下の手順を必須とします。

### 3.1 誰が変更できるか

- 設問の追加・削除・文言変更・配点変更は、リポジトリのメンテナ承認のもとで提案できます。
- 提案者は誰でも構いませんが、3.2 の根拠紐付けを満たさない提案はマージしません。

### 3.2 変更の根拠（リサーチゲートへの紐付けが必須）

- すべての設問変更は、`docs/research/evidence-matrix.csv` の根拠（evidence matrix）に紐付けてください。
- 新規・改訂された設問は、少なくとも1件の `source_id` を根拠として持つ必要があります。
- 根拠が evidence matrix に未登録の場合、まず matrix に行を追加（`source_id`、`title`、`source_category`、`credibility`、`claim`、`disposition` など）してから設問を変更します。
- 専有文言の流用リスクがある根拠は `copyright_risk` 列に記録し、`disposition` を `reject` か `adapt` とし、流用ではなく独自表現へ翻案します。

### 3.3 トレーサビリティ（設問ID → 根拠ソースID）

- 設問ごとに安定した question_id を維持します。
- PR 本文に「設問ID → 根拠ソースID」の対応表を記載します。
  - 例: `gov-01 → S-0007, S-0021`
- evidence matrix の `maps_to` 列に、対応する question_id または成熟度レベルを記録します。
- ID は削除より非推奨（deprecate）を優先し、過去レポートとの追跡性を保ちます。

### 3.4 レビュー手順（最低1名）

- 設問改訂PRは、提案者以外の最低1名のレビュー承認を必須とします。
- レビュー観点:
  - 根拠（evidence matrix の行）が妥当で、信頼度が記録されているか
  - 専有文言の流用がないか（2章の非交渉原則）
  - 日本語として自然で、B2B業務トーンを保っているか
  - 配点・閾値が `sqec.md` のスコアリング仕様と整合するか
  - 過去設問IDの追跡性が壊れていないか

## 4. 開発・検証手順

### 4.1 ローカル起動

`README.md` の手順に従ってください（Node、Python、または `docs/index.html` 直接オープン）。

### 4.2 必須チェック

PR を出す前に、以下をローカルで通してください。

- 主要JSファイルの構文チェック:
  ```bash
  node --check docs/js/app.js
  node --check docs/js/scoring.js
  node --check docs/data/questions.js
  ```
- テストの実行（テストが存在する場合）:
  ```bash
  node --test
  ```
- 受け入れ確認: `docs/acceptance-checklist.md` の該当項目を確認します。

## 5. コミット規約

Conventional Commits 形式を推奨します。

```
<type>(<scope>): <要約（日本語可）>
```

- type: `feat` / `fix` / `docs` / `refactor` / `test` / `chore` / `style`
- scope 例: `questions` / `scoring` / `app` / `ui` / `research` / `governance`
- 例:
  - `docs(questions): gov-03 の文言を evidence S-0042 に基づき修正`
  - `feat(scoring): カテゴリ重み設定を追加`

1コミット1論点を原則とします。

---

## 6. PRの出し方

1. `ma-garin/AIDD-Process-Next` をフォークし、作業ブランチを作成します。
2. 変更を小さくまとめ、4章のチェックを通します。
3. PR 本文に以下を記載します。
   - 変更概要と背景
   - 設問変更がある場合は「設問ID → 根拠ソースID」の対応表
   - 影響範囲（UI / スコアリング / データ / ドキュメント）
   - 実施した検証（`node --check`、`node --test`、UI確認）
4. 最低1名のレビュー承認を得てからマージします。
5. ユーザー影響のある変更は `CHANGELOG.md` の `[Unreleased]` に追記します。

---

## 7. 禁止事項の再掲

`agent.md §7` を参照してください。特に、外部送信・外部依存の追加、設問のUI直書き、未検証スコアの認証的表現、専有文言の流用は禁止です。
