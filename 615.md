# AIDD Process Next 改善提案（615）

対象システム: **AIDD Process Next（AI駆動開発 品質成熟度アセスメント）**
根拠ドキュメント: `agent.md`（運用マニュアル / リサーチゲート）, `sqec.md`（受け入れ・品質基準）
作成日: 2026-06-15

> 本提案は `sqec.md` の受け入れ基準・品質ゲート（QG-1〜QG-9）と `agent.md` のリサーチゲート（§5）を一次根拠とし、各提案に対応条項を明記する。

## 0. 全体所見（前提）

- 現状は**設計・リサーチゲート段階**（`agent.md §14` / `sqec.md §19`：300ソースのリサーチ未完了）。よって最優先は「実装」より「モデルの根拠化」と「v0.1の検証可能性」。
- 最大の差別化は `agent.md §3` の一文＝**AIエージェント自身が被評価者かつ利用者（self-audit protocol）**。提案はこの差別化を太らせる方向を優先する。
- v0.1制約（backend/auth/DB/AI API不要・静的・JSON駆動）を壊さない範囲を「v0.1」、それを超えるものは「v0.2+」と明記する。

| U1 | 結果画面＝レーダーチャート＋リスク優先のF字レイアウト | 高 | 中 |
| U2 | 0-3スコアに行動記述アンカー（BARS）をUIへ常時表示 | 高 | 中 |

- **U1 結果画面のレーダーチャート＋F字レイアウト**（優先度:高 / 難易度:中）
  論拠: `sqec.md §10.2`「Results screen prioritizes score, maturity, high risks, and next actions」。10診断領域（`agent.md §8`）はレーダー/バーで一覧比較が最も認知効率が高い（`sqec.md §17.3` 改善候補、ただしB2B診断の説得力に直結）。「同一カードの羅列禁止」（`§10.1`）への直接回答。

- **U2 0-3スコアに行動記述アンカー（BARS）**（優先度:高 / 難易度:中）
  論拠: `sqec.md F-004` は「0-3を選ぶ」だけで、評価者主観のばらつき（自己申告バイアス）を防げない。各設問の各レベルに具体的な振る舞い記述（例: 2=「定義され再現可能」`§8.1`）をUIに常時表示することで、`§8.3` の再現性（reproducible）と `§6` の説明可能性を担保。質問JSON `choices` をリッチ化（`§12.2`）。

- **U3 色のみに依存しない三重符号化**（優先度:中 / 難易度:低）
  論拠: `sqec.md §10.3 / §13`「Risk is not communicated by color alone」「Important actions depend on color alone = Fail」。色＋アイコン＋ラベルの3点セットで赤/橙/緑/青/灰のセマンティクスを表現。WCAGコントラスト（`§13`）も同時に満たす。低コストでアクセシビリティゲート（一般業務利用想定）を通過。

- **U4 カテゴリ別プログレス＋中断/再開**（優先度:中 / 難易度:中）
  論拠: 50問（`§15.1` / `§17.1`）は長く離脱リスクが高い。左ナビ（`§10.2`）＋カテゴリ進捗＋localStorage保存（`§17.3` 改善候補）で文脈を維持。v0.1の静的制約内で実装可（外部送信なし `§11.1`）。

- **U5 レポート画面の階層設計＋印刷/PDF対応**（優先度:中 / 難易度:中）
  論拠: `sqec.md §10.4`「clear hierarchy / Future PDF output is feasible」、`§17.3` print-friendly。経営層・QAリード・PMに「casualに見えない」（`§10.2`）コンサル級レポートが導入判断材料になる。CSSの`@media print`で低コスト対応可。

## 2. CX（カスタマーエクスペリエンス）（5件）

| # | 提案 | 優先度 | 難易度 |

- **C1 サンプル先行のゼロ摩擦体験**（優先度:高 / 難易度:低）
  論拠: B2B評価ツールは「触って価値を実感」までの距離が導入障壁。`sqec.md F-013`（サンプル読込）を**トップ画面の主導線**に置き、ログイン・登録なし（`§11.1`）で完成形レポートを即体験させる。`agent.md §4.1` の主要ユーザー（QAリード/AIテックリード）の評価コストを最小化。

- **C2 改善issueの直接エクスポート**（優先度:高 / 難易度:中）
  論拠: `agent.md §2-9`「Every diagnosis must lead to an action」、`sqec.md F-010`。診断を「点数で終わり」にせず、低スコア領域→改善issueドラフト（`§7.2` Fail: improvement issues not tied to low-scoring areas）をGitHub Issueテンプレ/Jira CSVで出力。実ワークフローへ接続し継続利用の動機を作る。

- **C3 スコアの根拠提示**（優先度:高 / 難易度:中）
  論拠: `sqec.md §2`「Evidence-backed / Explainable」、`agent.md §2-8`「connect to evidence」。総合レベルの算出根拠（どの回答が効いたか）と各設問の`evidenceRefs`（`§12.2`）を結果画面で開示。B2B顧客の信頼（`agent.md §4.1` 第三者検証会社含む）に直結。

- **C4 診断履歴＋差分**（優先度:中 / 難易度:中）
  論拠: `sqec.md §17.3` 改善候補（diagnosis history）。成熟度ツールの価値は「時間軸での改善可視化」。再取込可能なJSON（→機能面F3）を前提に、前回との delta を別セクション表示。`agent.md §8` Metrics & Continuous Improvement の自製品への適用。

- **C5 AIセルフチェックのワンクリックcopy＋エージェント別**（優先度:高 / 難易度:低）
  論拠: `sqec.md F-011 / §9`、`agent.md §4.2`。**唯一最大の差別化要素**を摩擦ゼロで提供。「Claude Code / Codex / Cursor に貼り付け可能」（`§9.2`）なプロンプトをcopyボタン付きで出力。低コストで差別化ゲート（QG-5）を満たす。

## 3. アルゴリズム（5件）

| # | 提案 | 優先度 | 難易度 |
|---|---|:--:|:--:|
| A1 | リスクベース加重スコアリング＋境界値の明確化・式の文書化 | 高 | 中 |
| A2 | 未回答/N/Aの明示扱いと分母調整（部分診断の妥当性） | 高 | 低 |
| A3 | 改善issue生成のルールベースロジック（低スコア→優先度付きアクション） | 高 | 中 |
| A4 | 成熟度判定の決定性・再現性（境界値テスト、reload不変） | 高 | 低 |
| A5 | 自己申告バイアス補正（証跡フラグによる加重ディスカウント） | 中 | 高 |

- **A1 加重スコアリング＋式の文書化**（優先度:高 / 難易度:中）
  論拠: `sqec.md §8.3`「Formula is documented」、qa-review規格の「総合点は単純平均にせず重み付け」。現行は0-3単純和（`§8.1-8.2`）。Security/Traceability等のリスク領域に加重し、`§8.2`の閾値（0-24…）を加重後スケールで再定義。**リサーチゲート後に重みを根拠化**（`§8.2` provisional）。

- **A2 未回答/N/Aの明示扱い**（優先度:高 / 難易度:低）
  論拠: `sqec.md §8.3`「Empty or incomplete answers are handled explicitly」（Fail: 未処理）。N/A（対象外）を許容し分母から除外、未回答は警告。部分診断でもレベルが破綻しないアルゴリズムを定義。`TC` の境界・混在ケースに対応。

- **A3 改善issue生成のルールベースロジック**（優先度:高 / 難易度:中）
  論拠: `agent.md §3` 差別化、`sqec.md F-010 / §8.3` Fail。v0.1はAI API不要（`agent.md §7`）なので**決定的ルールベース**で `riskIfLow → issueTemplate`（`§12.2`）を生成し、`スコア×重み`で優先度付け。非決定LLMに頼らず再現性を確保。

- **A4 判定の決定性・再現性**（優先度:高 / 難易度:低）
  論拠: `sqec.md §8.3`「reproducible by hand / Boundary scores are tested / Score changes unexpectedly on reload = Fail」。`TC-003`（全0→Lv.0）`TC-004`（全3→Lv.3）`TC-005`（手計算一致）を満たす純関数として採点を実装（状態非依存・イミュータブル）。Scoring Validity Gate（QG-4）の中核。

- **A5 自己申告バイアス補正**（優先度:中 / 難易度:高）
  論拠: `agent.md §7.3` リポジトリ証跡（CI/テスト/AGENTS.md有無）は機械検証可能で、自己申告（`§7.1/7.2`）より信頼度が高い。証跡裏付けのない高スコアを割引、または `evidenceRefs` 有無で信頼度メタを付与。`sqec.md §5.2` reliability分類の思想を採点へ適用。**v0.2+**（証跡取得＝機能面F1依存）。

## 6. 優先度 × 難易度 マトリクス（着手順の指針）

```
          難易度:低            難易度:中              難易度:高
優先度:高  C1 C5 F2 F4         U1 U2 C2 C3 A1        X1
          A2 A4 X3             A3 X2 X4
優先度:中  U3 F3               U4 U5 C4 X5            F1 A5
優先度:低                      F5
```

### 推奨ロードマップ
1. **土台（ブロッカー）を先に**: `X1`（リサーチゲート運用基盤）＝これが未完だとモデルが根拠化されず後工程が手戻り（`agent.md §5` exit criteria）。並行で `X2`（命名/著作権CI）。
2. **クイックウィン（高優先・低難易度）**: `C1`（サンプル先行）`C5`/`F2`（AIセルフチェック＝差別化）`F4`（質問スキーマ）`A2`/`A4`（採点の決定性）`X3`（セキュリティ）。少コストでQG-4/5/7に効く。
3. **製品の説得力**: `U1`（レーダー＋リスク優先）`U2`（BARS）`C2`/`C3`（アクション接続・根拠提示）`A1`/`A3`（加重・issue生成）。QG-1後にA1の重みを根拠化。
4. **品質の固定**: `X4`（テスト証跡）`X5`（受け入れゲート）→ `sqec.md §18` Definition of Done 達成。
5. **v0.2+**: `F1`（リポジトリ証跡）→ `A5`（自己申告バイアス補正）で差別化を更に強化。

> 注: v0.1制約（no backend/auth/DB/AI API）を破るのは `F1`/`A5` のみ。これらは v0.2+ に隔離し、v0.1は静的・JSON駆動・決定的採点を堅持する（`agent.md §7`）。
