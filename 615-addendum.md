## 1. UI/UX（5＋1件）

| # | 提案 | 優先度 | 難易度 |
|---|---|:--:|:--:|
| UX1 | キーボード操作＋フォーカス管理 | 高 | 低 |
| UX2 | 空状態・エラー状態・未完了ガードのUI | 高 | 低 |
| UX3 | 提出前レビュー画面（回答サマリ） | 中 | 中 |
| UX4 | ダークモード対応 | 中 | 低 |
| UX5 | カテゴリ冒頭のコンテキスト提示＋所要時間目安 | 中 | 低 |
| UX6 | 結果レーダーのインタラクティブ化（任意） | 低 | 中 |

- **UX1 キーボード操作＋フォーカス管理**（高 / 低）
  数字キー1–4で回答・N でN/A・←→で前後設問・Enterで次へ。画面遷移時に見出しへ `focus()` を移し `aria-live` で通知。
  論拠: `app.js` に `keydown` ハンドラ皆無（事実2）。50問の逐次回答は反復操作が重く、キーボード/スクリーンリーダー利用者の操作経路が無い。
  **615差分**: U3=色の三重符号化、U4=進捗ナビ。キーボード入力導線・フォーカス遷移は未カバー。

- **UX2 空状態・エラー状態・未完了ガードのUI**（高 / 低）
  questions.js ロード失敗時の代替表示、未回答のまま「結果を見る」時の確認（回答率と影響を明示）、ゼロ回答時の結果画面の扱いを定義。
  論拠: ロード失敗時のフォールバック未実装、`calcTotalScore` は未回答を黙って除外し誤誘導しうる（事実7）。
  **615差分**: F4=JSONスキーマ検証（ロジック層）。ユーザー向けの空/エラー/未完了画面パターンは未カバー。

- **UX3 提出前レビュー画面（回答サマリ）**（中 / 中）
  全50回答を一覧表示し、低スコア/未回答/N/Aを色分け、その場で修正→結果へ。
  論拠: 現状は前/次の逐次ナビのみ（`app.js`）。回答全体を俯瞰し見直す手段が無い。
  **615差分**: U4=カテゴリ進捗。提出前の全回答俯瞰レビューは別物。

- **UX4 ダークモード対応**（中 / 低）
  `prefers-color-scheme` + `tokens.css` のCSS変数を二系統化。手動トグルも任意。
  論拠: tokens は変数化済で低コスト、`prefers-color-scheme` 未対応（事実4）。B2B長時間利用の眼精疲労低減。
  **615差分**: U3=色のセマンティクスのみ。テーマ切替は未カバー。

- **UX5 カテゴリ冒頭のコンテキスト提示＋所要時間目安**（中 / 低）
  各領域の設問群に入る前に「目的/なぜ重要か」（`agent.md §8` の領域趣旨）と残り設問数・目安時間を提示。
  論拠: questions.js にカテゴリ趣旨はあるが回答前に提示されず、自己申告の文脈設定が弱い。
  **615差分**: U2=設問内BARS。カテゴリ単位の事前コンテキスト/時間目安は別。

- **UX6（任意）結果レーダーのインタラクティブ化**（低 / 中）
  ホバーで数値、クリックでそのカテゴリの低スコア設問へジャンプ。`state.radarChart`（未使用デッドフィールド・事実1）の整理も同時に。
  論拠: `renderRadarChart`（`app.js:288`）は静的SVG。
  **615差分**: U1=レーダー描画の存在。操作性付与は別。

## 2. アルゴリズム（5＋1件）

| # | 提案 | 優先度 | 難易度 |
|---|---|:--:|:--:|
| AL1 | 回答率ベースの信頼度＋レベル確定の保留 | 高 | 低 |
| AL2 | 次レベルまでの逆算（gap-to-next-level） | 高 | 中 |
| AL3 | リスク優先順位の複合スコア化＋件数の動的化 | 高 | 中 |
| AL4 | 回答整合性・矛盾検出 | 中 | 中 |
| AL5 | 設問内重み（カテゴリ内の設問重要度差） | 中 | 中 |
| AL6 | 感度分析／スコア安定性の開示（任意） | 低 | 中 |

- **AL1 回答率ベースの信頼度＋レベル確定の保留**（高 / 低）
  回答率が閾値未満なら総合レベルを「暫定」と明示し、信頼度メタ（completion%・有効カテゴリ数）を併記。
  論拠: `calcTotalScore`（`scoring.js:24-36`）は有効回答のみ平均 → 数問だけ満点回答で Lv.3 と誤表示しうる（事実7）。
  **615差分**: A2=N/A分母除外、A4=決定性。低回答率時の信頼度/確定保留ロジックは未カバー。

- **AL2 次レベルまでの逆算（gap-to-next-level）**（高 / 中）
  「あと何点でLv上昇」「どの低スコア設問を+1すれば最も効くか」を算出し結果/レポートへ。
  論拠: `determineLevel`（`scoring.js:38`）は境界判定のみ。最小改善→最大効果の提示で診断を行動へ接続（`agent.md §2-9`）。
  **615差分**: C3=根拠提示。next-level逆算（最適改善ターゲット）は別。

- **AL3 リスク優先順位の複合スコア化＋件数の動的化**（高 / 中）
  並び順を「カテゴリ重要度×スコア×（将来）証跡」に、issue件数は固定8件でなく閾値で動的に。
  論拠: `getRisks` は score昇順→categoryId（`scoring.js:89`）、`getIssues` は `slice(0,8)` 固定（`scoring.js:93`）でカテゴリ重要度を反映しない（事実9）。
  **615差分**: A1=総合点のカテゴリ加重。リスク並び順とissue件数ロジックは別。

- **AL4 回答整合性・矛盾検出**（中 / 中 / v0.1可）
  論理的に起きにくい組合せ（例: 自己監査C9が高いのにAI成果物レビューC4が0）を検出し再確認を促す。
  論拠: 自己申告バイアスへの一次防衛線。純関数採点（`scoring.js`）の出力に対しルールで判定可能。
  **615差分**: A5=証跡による割引（v0.2+・証跡取得依存）。回答内の整合性検査はv0.1で実装可能な別施策。

- **AL5 設問内重み（カテゴリ内の設問重要度差）**（中 / 中）
  `calcCategoryScore` の単純平均（`scoring.js:21`）を、設問重み付き平均へ拡張（リサーチゲート後に重みを根拠化）。
  論拠: 設問の重要度が均一前提のため、致命的設問が薄まる（事実8）。
  **615差分**: A1=カテゴリ**間**重み。カテゴリ**内**設問重みは未カバー。

- **AL6（任意）感度分析／スコア安定性の開示**（低 / 中）
  1設問変更でレベルが変わる「閾値付近」を機械的に算出し開示、過信を防ぐ。
  論拠: 純関数採点なので全設問の摂動を機械計算可能。
  **615差分**: 615に該当なし。

- **FT3 PWA化（manifest＋Service Worker）**（中 / 中 / v0.1可）

- **FT4 設問ごとの根拠メモ機能**（中 / 中）
  各回答に証跡メモ（任意）を残しレポート/issueへ反映。「事実/推定/未確認を分ける」思想を入力UXへ。
  論拠: `agent.md` の証跡文化を入力段階で支援。`escapeHtml`（`scoring.js:112`）で安全に格納可。

  論拠: 一般業務利用想定でWCAG必要だが a11y専用の自動チェックが無い。
  **615差分**: X4=機能テスト全般。a11y専用CIは別の具体化。
